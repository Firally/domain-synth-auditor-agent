# Domain Synthetic Data Auditor Agent (PoC)

PoC агентной системы для автоматизации контроля качества синтетических изображений (prompt → image) и подготовки датасетов для задач компьютерного зрения в доменно-специфичных сценариях (склад, магазин, лифт, производство и др.).

## 1) Что за задача, для кого и какая боль сейчас

### Для кого
- команды Computer Vision / ML, которые решают доменные задачи (детекция/сегментация/классификация) и используют или планируют использовать синтетические данные;
- инженеры/исследователи, которым нужно быстро получать пригодные данные для обучения и сравнивать подходы.

### Боль
В доменных задачах узкое место — данные и контроль качества синтетики:
- генерация может быть "красивой", но нерелевантной домену (не тот интерьер/объекты/контекст);
- промпты и параметры приходится долго подбирать вручную;
- сложно системно отфильтровать артефакты и "плохие" примеры;
- есть риск попадания чувствительных данных (PII) или запрещённого контента в датасет;
- отсутствует воспроизводимый процесс: почему пример принят/отклонён, какие итерации промпта сработали, как менялось качество.

### Цель PoC
Сократить ручной труд на QC синтетики и ускорить получение пригодного датасета за счет агентного контура:
**генерация → аудит качества → accept/reject/needs-review → рекомендации по улучшению промпта → логирование и память.**

## 2) Что именно сделает PoC на демо

Демо показывает end-to-end цикл на 1–2 доменных задачах (пример: “наклейки/маркировки”, “кабели/провода”, “повреждения”, “объекты зоны”):

1. Пользователь задаёт:
   - краткое описание домена и задачи,
   - базовый prompt и ограничения (что должно/не должно быть),
   - бюджет (кол-во итераций/картинок).

2. Система:
   - генерирует N изображений через API генератора;
   - проводит аудит качества:
     - соответствие промпту (VLM-оценка + объяснение),
     - доменная релевантность (правила домена + сравнение с референсами),
     - техническое качество (IQA/blur/spoofing),
     - safety/PII-check (лица/документы/телефоны/QR и т.п. → reject/review);
   - для каждого изображения принимает решение:
     - **ACCEPT** (можно добавлять в датасет),
     - **REJECT** (бракуем),
     - **NEEDS_REVIEW** (отправить на ручную проверку/разметку);
   - предлагает улучшения промпта/параметров и запускает следующую итерацию;
   - сохраняет эксперимент (промпты, параметры, скоры, причины решений) в трекер и в память системы.

3. Выход демо:
   - отчёт по итерациям (accept/reject/review + причины),
   - финальный “recipe” промпта/настроек,
   - версия датасета с принятыми изображениями + список на ручную проверку.

### Метрики успеха PoC
- acceptance rate (доля ACCEPT среди сгенерированных);
- снижение ручной проверки (доля, не ушедшая в NEEDS_REVIEW);
- число итераций до достижения целевого качества;
- согласие с человеком на сэмпле (корреляция решений аудитора и ручного QA).

## 3) Что НЕ делает PoC (out-of-scope)

- Не является продовой системой и не выводит модели в production.
- Не гарантирует улучшение downstream-метрик моделей (это опциональная проверка).
- Не заменяет ручную проверку полностью: human-in-the-loop остаётся частью процесса.
- Не обеспечивает промышленную безопасность — только базовые политики safety/PII и безопасного запуска инструментов.

## Quick Start

### Требования

- Python 3.11+
- API-ключ [OpenRouter](https://openrouter.ai/keys) (для VLM и генерации изображений)
- MySQL (опционально — только для `--source db`)

### Установка

```bash
# Клонировать репозиторий
git clone https://github.com/<your-username>/domain-synth-auditor-agent.git
cd domain-synth-auditor-agent

# Создать виртуальное окружение и установить зависимости
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# Настроить переменные окружения
cp .env.example .env
# Заполнить OPENROUTER_API_KEY в .env
```

### Запуск

**Smoke test** — проверка подключения и e2e пайплайна на одном изображении:

```bash
# Только проверка подключения к API
python scripts/smoke_test.py --connectivity-only

# Полный smoke test (генерация + аудит + решение)
python scripts/smoke_test.py --project wb_pvz
```

**Batch processing** — обработка пакета изображений из CSV:

```bash
python scripts/run_batch.py \
  --csv data.csv \
  --project tv_settings \
  --limit 5
```

**Batch processing из MySQL** (требуется настроенная БД):

```bash
# Инициализировать схему БД
mysql -u root < scripts/init_db.sql

# Запуск с natural language запросом
python scripts/run_batch.py \
  --source db \
  --project tv_settings \
  --request "Сделай 10 картинок телевизоров с экранами настроек"
```

**Evals** — запуск evaluation на golden set:

```bash
# Аудит-eval (agreement с human labels)
python scripts/run_evals.py --verbose

# IntentResolver eval (точность определения типа объекта)
python scripts/run_intent_evals.py
```

### Структура проекта

```
src/auditor/          # Ядро системы
  pipeline.py         # DVF-петля: Draft → Verify → Fix
  audit_stage.py      # VLM-аудит изображения
  decision_engine.py  # ACCEPT / REJECT / NEEDS_REVIEW
  model_gateway.py    # Обёртка над OpenRouter API
  prompt_builder.py   # Генерация промптов из domain spec
  prompt_improver.py  # LLM-улучшение промпта на Fix-шаге
  knowledge_base.py   # Retrieval из reference annotations
  memory_store.py     # Межитерационная память агента
  intent_resolver.py  # NL-запрос → тип объекта из БД
  db_loader.py        # MySQL: загрузка изображений по типу
  image_loader.py     # CSV: загрузка изображений по URL
  domain_spec.py      # Парсинг domain.yaml
  config.py           # Конфигурация (env vars)
  experiment_store.py # Сохранение артефактов прогонов

scripts/              # Точки входа
projects/             # Доменные конфигурации (domain.yaml + каталоги)
evals/                # Golden sets для evaluation
docs/                 # Архитектура, спеки, диаграммы
```

### Доменные проекты

Каждый проект в `projects/` содержит `domain.yaml` с конфигурацией домена.
Готовые примеры:
- `wb_pvz` — ПВЗ (генерация сцен)
- `tv_settings` — замена контента на экранах ТВ (edit mode)
- `broken_playgrounds` — детекция повреждений на детских площадках
