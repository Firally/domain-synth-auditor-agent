# Product Proposal: Domain Synthetic Data Auditor Agent (PoC)

## 1) Обоснование идеи: прикладная задача

### Контекст
Во многих задачах компьютерного зрения данные являются основным ограничением: редкие события, дисбаланс классов, высокая стоимость разметки, неоднородные условия съёмки (ракурсы, освещение, устройства), а также дрейф данных во времени.  
Синтетические данные (prompt → image) позволяют ускорить получение обучающих примеров, однако на практике возникает новая "узкая точка" — контроль качества синтетики.

### Проблема (pain points)
1. **Ручной QC синтетики занимает много времени.**  
   Чтобы синтетические изображения были пригодны для обучения, их приходится вручную отбирать, фильтровать артефакты, проверять соответствие задаче и домену.
2. **Сложно понять, почему генерация "не работает".**  
   Причины могут быть разными: промпт сформулирован слабо/противоречиво, генератор рисует нерелевантный фон, объект отсутствует или искажен, присутствуют лишние объекты.
3. **Отсутствует воспроизводимость.**  
   Часто не сохраняется, какие промпты/параметры использовались, почему пример приняли/отклонили, какие итерации дали лучший результат.
4. **Риски безопасности и комплаенса.**  
   Генерации или реальные изображения могут содержать PII (лица, документы, телефоны, QR/баркоды), что требует политики фильтрации/анонимизации.
5. **Невозможность быстро перенести подход на новый домен.**  
   Для каждого домена приходится заново “на глаз” формировать критерии качества и шаблоны промптов.

### Предлагаемое решение (PoC)
PoC агентной системы, которая автоматизирует аудит синтетических изображений и итеративную оптимизацию генерации:
**генерация → автоматический аудит качества → решение accept/reject/needs-review → рекомендации по правке промпта/параметров → логирование и память**.

Система должна быть переносимой на разные домены через “Domain Pack”:
- правила домена (rubric),
- референсные изображения,
- шаблоны промптов/негативных промптов,
- список целевых объектов/запрещённых сущностей.

## 2) Цель и метрики успеха

### Цель проекта (продуктовая)
Сократить время и ручные усилия на получение синтетического датасета пригодного качества для CV-задач в доменно-специфичных сценариях.

### Продуктовые метрики
- **Manual QA Reduction**: доля изображений, которые не требуют ручной проверки (не попали в `NEEDS_REVIEW`).
- **Acceptance Rate**: доля изображений, принятых в датасет (`ACCEPT`) от общего числа сгенерированных.
- **Iterations to Target**: среднее число итераций промпта/параметров до достижения целевого acceptance rate.
- **Time-to-Dataset** (если есть измерение): время до получения N принятых изображений для датасета.

### Агентские метрики (качество решений агента)
- **Human Agreement**: согласие с ручным QA на контрольном сэмпле (например, accuracy/F1 по классам `accept/reject/review`).
- **Reason Quality**: доля решений, где агент выдает корректную и проверяемую причину (структурированный violation + evidence).
- **Prompt Improvement Effect**: улучшение метрик качества после оптимизации промпта агентом (например, рост acceptance rate vs baseline prompt).
- **Policy Compliance**: отсутствие пропусков PII/запрещённого контента в принятых изображениях (0 критических инцидентов в PoC).

### Технические метрики
- **p95 latency на аудит 1 изображения** (без генерации) — целевое значение задается бюджетом PoC.
- **Throughput batch-audit**: количество изображений/минуту при пакетной обработке.
- **Stability**: устойчивость скоринга на разных seeds (вариативность оценок).
- **Reproducibility**: полный лог эксперимента (prompt, параметры, версии тулов, результаты) восстанавливается по run-id.

## 3) Потенциальные сценарии использования (включая edge-cases)

### Сценарий A: Быстрый запуск синтетики для новой доменной задачи
**Вход:** Domain Pack (референсы + правила) и базовый prompt.  
**Шаги:**
1) генерация N изображений;  
2) автоматический аудит (соответствие, доменная релевантность, IQA, safety);  
3) triage: accept/reject/review;  
4) агент предлагает улучшения промпта и повторяет цикл.  
**Выход:** версия датасета + отчет причин/статистики.

**Edge-cases:**
- промпт конфликтный ("без текста", но требуется "стикер с текстом") → агент выявляет конфликт и предлагает исправление;
- генератор стабильно уходит в неправильный стиль → агент включает ограничения (negative prompts/стиль) и сигналит про лимит итераций.

### Сценарий B: Доменно-нерелевантная, но формально "правильная" генерация
Пример: объект присутствует, но фон/контекст не соответствует домену.  
**Поведение:** доменный агент помечает `NEEDS_REVIEW` или `REJECT` и объясняет, какие доменные правила нарушены.

### Сценарий C: Сложный объект / мелкие детали (стикеры, провода)
**Риск:** VLM ошибается на мелких деталях, OCR шумит.  
**Поведение:** система:
- локализует область интереса (опционально) и ограничивает OCR/проверки;
- при низкой уверенности переводит в `NEEDS_REVIEW` вместо жесткого reject.

### Сценарий D: Safety/PII
**Риск:** лица, документы, номера телефонов, QR/баркоды и т.д.  
**Поведение:** политика:
- auto-reject или auto-redact (в PoC можно auto-reject),
- логирование только ссылок на защищённое хранилище,
- отдельный отчет по safety-отказам.

### Сценарий E: Дрейф правил или смена домена
**Вход:** новый Domain Pack.  
**Поведение:** агент начинает с “нулевой” конфигурации домена, но может переиспользовать общие паттерны промптинга и типовые причины брака.

## 4) Ограничения (технические и операционные)

### Технические ограничения (предварительные)
- **Latency**:
  - аудит одного изображения включает несколько тулов (VLM, IQA, OCR, safety) и может быть дорогим;
  - в PoC допустима пакетная обработка; требования к p95 latency фиксируются по среде запуска.
- **SLO/SLA** (для PoC — ориентиры):
  - стабильность работы пайплайна (ошибки API, таймауты) — предусмотрены retry и failover в `NEEDS_REVIEW`.
- **Качество VLM/OCR/детекторов**:
  - возможны ошибки, особенно на мелких объектах; PoC учитывает это через пороги уверенности и human-in-loop.
- **Воспроизводимость**:
  - обязательны сохранение seeds/параметров генерации, версии моделей/тулов, конфиг домена.

### Операционные ограничения
- **Бюджет на генерацию и аудит**:
  - лимит на число итераций (например, K итераций и N изображений на итерацию);
  - ограничение на стоимость вызовов внешних API (VLM, генератор).
- **Human-in-the-loop**:
  - часть кейсов неизбежно уходит на ручную проверку (`NEEDS_REVIEW`);
  - PoC фиксирует долю таких кейсов как одну из метрик успеха.
- **Безопасность**:
  - ограничение на хранение/логирование изображений, содержащих PII;
  - политика allowlist для инструментов и ограничение действий агента (например, запрет на массовые операции без подтверждения).
- **Scope PoC**:
  - демонстрация на 1–2 задачах и одном Domain Pack;
  - downstream-обучение модели — опционально и зависит от наличия инфраструктуры.

## 5) Архитектурный набросок (модули и интеграции)

### 5.1. Выбранный паттерн
PoC строится как **agentic workflow**: общий пайплайн фиксирован (генерация → аудит → решение), но внутри шагов используется агентная логика (ветвления, подбор инструментов, итерации). Центральный паттерн — **Evaluator–Optimizer**: один контур оценивает качество синтетики, второй предлагает улучшения промпта/параметров и перезапускает генерацию в рамках бюджета.

### 5.2. Основные модули

**1) API/CLI слой (Entry Point)**
- Принимает `Task Spec`:
  - описание домена/сцены (коротко),
  - целевые объекты/ограничения (must-have / must-not-have),
  - базовый prompt (+ optional negative prompt),
  - бюджет (N изображений, K итераций), пороги принятия.
- Выдаёт отчёт по прогону и артефакты (ссылки на датасет/логи).

**2) Orchestrator (менеджер пайплайна)**
- Управляет шагами пайплайна, бюджетами, ретраями, таймаутами.
- Ведёт состояние запуска (итерации, результаты проверок, решения).
- Запускает проверки **параллельно** (где возможно) и агрегирует результат.
- Решает ветвления:
  - `ACCEPT` → сохранить в датасет
  - `REJECT` → отбраковать
  - `NEEDS_REVIEW` → отправить в human-in-the-loop

**3) Knowledge Layer: Domain Pack + Retrieval (RAG)**
Набор доменных знаний, который делает систему переносимой:
- `domain_rules` (rubric): правила релевантности и причины отказов,
- `reference_images`: примеры “good/bad/edge”,
- `prompt_templates`: шаблоны промптов/negative prompts.
Retrieval вытаскивает релевантные правила/шаблоны под конкретную задачу.

**4) Audit Engine (набор проверок как tools/workers)**
Каждая проверка — отдельный инструмент (или worker), возвращает структурированный результат:
- **Prompt Compliance (VLM/LLM-as-judge):**
  - соответствие промпту + объяснение причин (evidence).
- **Domain Relevance (rules + references + VLM):**
  - похоже ли на целевой домен/сцену, какие правила нарушены.
- **Technical Quality (IQA/spoofing):**
  - смаз/пересвет/артефакты/подозрение на “невалидное” изображение.
- **Safety/PII Check:**
  - лица/документы/телефоны/QR/баркоды → политика reject/review.
Опционально (для задач с мелкими объектами): локализация области интереса (open-vocab det) и/или маски (SAM) для точечного OCR/проверок.

**5) Decision & Policy Layer (Aggregator)**
- Агрегирует сигналы Audit Engine.
- Применяет правила принятия решения (thresholds + приоритеты):
  - hard-fail (например PII) → `REJECT`
  - противоречивые/пограничные сигналы → `NEEDS_REVIEW`
  - все проверки пройдены → `ACCEPT`
- Формирует финальный “verdict” + список причин/нарушений.

**6) Prompt Optimizer (LLM)**
- На основании причин `REJECT/REVIEW` предлагает:
  - уточнения/ограничения промпта,
  - negative prompt,
  - изменения параметров генерации (если допускается).
- Запускается только при наличии бюджета итераций (K) и при соблюдении guardrails.

**7) Memory (сквозная память)**
- Память “рецептов”: какие промпты/шаблоны и параметры давали высокий acceptance rate для данного домена/задачи.
- Память “ошибок”: типовые причины отказов, примеры edge-case.
Реализация в PoC может быть простой:
- структурированное хранилище (JSON/SQLite) + опционально векторный индекс для поиска похожих кейсов.

**8) Storage & Experiment Tracking**
- Хранилище изображений (object storage/файлы): `raw_generated/`, `accepted/`, `rejected/`, `needs_review/`.
- Метаданные: результаты проверок, промпты, seeds, версии domain pack, решения (DB/JSON).
- Трекинг экспериментов (например ClearML): параметры, метрики, артефакты, отчёты.

**9) Human-in-the-loop интеграция (опционально, но желательно)**
- Экспорт `NEEDS_REVIEW` в разметку/ревью (например Label Studio batch).
- Возврат статуса (approved/rejected) для дальнейшего улучшения правил и памяти.

### 5.3. Интеграции (Tools/API)
Минимальные интеграции для PoC:
- API генератора изображений (diffusion/adapter),
- VLM API (captioning + judge),
- IQA/spoofing сервис/модель,
- OCR (для текста/стикеров/PII),
- трекинг экспериментов (ClearML или аналог),
- хранилище артефактов (файлы/S3).

Опциональные интеграции:
- open-vocabulary детектор (для локализации объектов),
- SAM/segmentation tool (для масок и ROI),
- Label Studio API (human review).

### 5.4. Поток данных (end-to-end)
1) `Task Spec` → Orchestrator  
2) Orchestrator → Generator API → N изображений  
3) На каждое изображение параллельно: IQA + Safety/PII + VLM checks (+ domain rules retrieval)  
4) Aggregator → `ACCEPT/REJECT/NEEDS_REVIEW` + причины  
5) Логирование (experiment tracker) + сохранение артефактов (storage)  
6) Если бюджет позволяет: Prompt Optimizer → новая итерация генерации  
7) Результат: отчёт, версия “принятой” синтетики, список на ручной review.

### 5.5. Guardrails и failover (на уровне архитектуры)
- Pre-check: валидация входного `Task Spec`, лимиты бюджета, allowlist инструментов.
- Runtime: таймауты/ретраи на внешних API, при сбое — перевод в `NEEDS_REVIEW`.
- Post-check: PII policy (reject/редакция), запрет логирования чувствительных артефактов в открытые логи.

## 6) Data flow и границы делегирования LLM/Agent

### 6.1. Потенциальный data flow (end-to-end)

**Входы**
- `Task Spec`:
  - домен/контекст (кратко),
  - цель (что должно быть на синтетике),
  - must-have / must-not-have объекты/атрибуты,
  - базовый prompt (+ optional negative prompt),
  - бюджет (K итераций, N изображений/итерацию), пороги решений,
  - политика safety (например: PII → reject).
- `Domain Pack`:
  - правила/рубрика качества (domain_rules),
  - reference_images (good/bad/edge),
  - prompt_templates.

**Шаг 0 — Валидация**
- проверка формата Task Spec, бюджетов, allowlist инструментов;
- нормализация: выделение ключевых объектов/ограничений.

**Шаг 1 — Генерация**
- генератор создаёт N изображений, сохраняются:
  - image_uri + metadata (seed, model_version, cfg/steps, prompt version).

**Шаг 2 — Параллельный аудит каждого изображения**
2.1 Technical quality: IQA/spoofing → `quality_score` + `fail_reasons`  
2.2 Safety/PII scan: лица/документы/телефоны/QR → `pii_flags` + `policy_action`  
2.3 Prompt compliance (VLM/LLM judge): соответствие промпту → `compliance_score` + `evidence`  
2.4 Domain relevance (rules + references + VLM): доменная релевантность → `domain_score` + `violations`

**Шаг 3 — Aggregation & Decision**
- Decision layer агрегирует сигналы и выдаёт:
  - `ACCEPT` / `REJECT` / `NEEDS_REVIEW`
  - структурированные причины (violations + evidence)
  - confidence / uncertainty

**Шаг 4 — Side effects**
- `ACCEPT`: сохранить в датасет (dataset_versioning), добавить теги/провенанс
- `REJECT`: сохранить только метаданные/причины (безопасная политика хранения)
- `NEEDS_REVIEW`: отправить батч в human review / разметку (Label Studio batch)

**Шаг 5 — Оптимизация промпта (итерации)**
- если бюджет не исчерпан:
  - Prompt Optimizer формирует исправления промпта/negative prompt/параметров
  - запускается следующая итерация (Шаг 1)

**Шаг 6 — Итоговый отчёт**
- статистика accept/reject/review
- топ причин reject
- “лучший recipe” промпта/параметров
- ссылки на артефакты и трекинг (ClearML/лог)

### 6.2. Что можно делегировать LLM/Agent (и почему)

**Можно (хорошие зоны для LLM/агента)**
1) **Интерпретация Task Spec и построение плана итераций**
- разбор требований, выделение must-have/must-not-have, сбор релевантных правил из Domain Pack.
2) **VLM/LLM-as-judge проверки**
- соответствие промпту (объяснимые причины),
- доменная релевантность по рубрике и референсам.
3) **Генерация рекомендаций**
- улучшение промпта и negative prompt,
- предложения по параметрам генерации (в рамках allowlist).
4) **Триаж и маршрутизация**
- решение `ACCEPT/REJECT/NEEDS_REVIEW` при наличии структурированных сигналов,
- формирование батчей на ручной review и коротких инструкций для человека.
5) **Агрегация и объяснения**
- превращение “сырых” результатов инструментов в понятный отчёт: что не так и что попробовать.

### 6.3. Что НЕ стоит делегировать LLM/Agent (или только с ограничениями)

**Нельзя / нежелательно полностью делегировать**
1) **Жёсткие safety-решения без правил**
- PII/комплаенс решения должны опираться на детерминированные политики (rule-based) и/или специализированные детекторы, а LLM может лишь объяснять и маршрутизировать.
2) **Необратимые действия**
- массовая генерация без лимитов, удаление данных, публикация артефактов, любая “опасная” операция → только через строгие лимиты + подтверждение.
3) **Единоличный “финальный” вердикт при низкой уверенности**
- пограничные кейсы (низкая уверенность VLM, конфликт сигналов, мелкие детали) → `NEEDS_REVIEW` и human-in-loop.
4) **Оценка качества мелких объектов как единственный источник истины**
- для стикеров/проводов VLM может ошибаться; нужны дополнительные сигналы (OCR, локализация ROI, качество изображения) и политика “если сомнительно — review”.

### 6.4. Human-in-the-loop: где человек обязателен

**Обязательные точки участия человека**
- утверждение первоначального `Task Spec` и доменных правил (Domain Pack) для нового домена;
- выборочная проверка качества синтетики (audit sampling), особенно на старте;
- рассмотрение `NEEDS_REVIEW` (сложные/пограничные/конфликтные кейсы);
- утверждение политики хранения/PII (что логируем, что удаляем);
- (если интегрировать дальше) approval на использование синтетики для обучения/релиза.

### 6.5. Guardrails (чтобы агент не “сломал” процесс)
- Budget guards: лимит K итераций и N изображений/итерацию, стоп при превышении.
- Tool allowlist: агент может вызывать только разрешённые инструменты и только с ограниченными параметрами.
- Decision guards: hard-fail правила (PII → reject), сомнение → review.
- Logging policy: чувствительные данные не пишем в открытые логи, только ссылки на защищённое хранилище.

## Итог PoC (что считаем "успешным")
PoC считается успешным, если:
- агент стабильно формирует triage (`accept/reject/review`) с объяснениями;
- достигается заметное сокращение ручной проверки и рост качества/релевантности синтетики по выбранным метрикам;
- воспроизводимость и логирование позволяют повторить эксперимент и понять, какие изменения улучшили результат;
- соблюдены базовые safety/PII политики (нет критических пропусков).
